# Code Review — 2026-03-03

## Scope
- FastAPI backend (`backend/app`)
- Frontend auth/session flow (`frontend/auth.js`)
- Test readiness (`backend/tests`)

## Executive Summary
Проект имеет рабочую базовую архитектуру (разделение API/сервисов/моделей, ролевой доступ, health-check), но содержит несколько критичных рисков безопасности и надежности, которые нужно закрыть перед production.

## Findings

### 1) Критично: небезопасные значения по умолчанию в конфигурации
**Severity:** Critical  
**Where:** `backend/app/config.py`

В коде зашиты дефолтные значения для `jwt_secret`, `superuser_login`, `superuser_password`, а также URL БД. Это создает риск компрометации при любом деплое, где `.env` не переопределен корректно.

**Recommendation:**
1. Убрать секреты/пароли из дефолтов (оставить пустые значения и валидировать на старте).
2. Падать при запуске, если `jwt_secret` не задан или равен небезопасному шаблону.
3. Отключить автосоздание суперпользователя по дефолту или требовать одноразовый bootstrap-флаг.

---

### 2) Критично: автоматические DDL-миграции в runtime
**Severity:** Critical  
**Where:** `backend/app/main.py` (`ensure_columns_and_enum` + DDL в `lifespan`)

Приложение выполняет `ALTER TABLE`, `CREATE TABLE`, `ALTER TYPE` при старте. Это опасно для production (гонки при многопроцессном старте, непредсказуемые падения, отсутствие контролируемой истории миграций).

**Recommendation:**
1. Перенести schema evolution в Alembic-миграции.
2. Оставить в приложении только проверки доступности БД.
3. Сделать отдельный этап deploy: `alembic upgrade head`.

---

### 3) Высокий риск: CORS fallback на `*`
**Severity:** High  
**Where:** `backend/app/main.py`

При пустом `CORS_ORIGINS` выставляется `allow_origins=["*"]` и `allow_credentials=True`. Для production это слишком широкая политика и потенциальный источник ошибок/утечек.

**Recommendation:**
1. Для production запретить wildcard.
2. Ввести fail-fast при пустом списке разрешенных origins.
3. Оставить wildcard только в явном dev-режиме.

---

### 4) Высокий риск: токен в `localStorage`
**Severity:** High  
**Where:** `frontend/auth.js`

JWT хранится в `localStorage`. При XSS токен легко извлекается и используется злоумышленником.

**Recommendation:**
1. Перейти на `HttpOnly` + `Secure` cookie (с корректным CSRF-паттерном).
2. Добавить CSP и базовую XSS-hardening политику.

---

### 5) Средний риск: отсутствует ограничение попыток логина
**Severity:** Medium  
**Where:** `backend/app/api/auth.py`

Нет rate limiting / lockout политики для `/auth/login`, что позволяет brute-force атаки.

**Recommendation:**
1. Ввести rate limit на IP + login.
2. Добавить экспоненциальную задержку/временную блокировку.
3. Логировать подозрительную активность и алертить.

---

### 6) Средний риск: неготовность тестового окружения
**Severity:** Medium  
**Where:** `backend/tests`

Тесты не запускаются в текущем окружении без `httpx` (ошибка импорта `starlette.testclient`). Это снижает стабильность CI и скорость регрессионной проверки.

**Recommendation:**
1. Зафиксировать и установить test-dependencies (`httpx`, `pytest-asyncio` при необходимости).
2. Добавить отдельный lock/requirements-dev и CI job на обязательный прогон тестов.

## Positive Notes
- Четкое разделение слоев в backend (`api`, `services`, `models`).
- Есть role-based access checks и централизованная логика прав.
- Присутствует `/health` endpoint и базовая обработка ошибок.

## Suggested Priority Plan
1. **P0:** Удалить небезопасные дефолты (`jwt_secret`, superuser creds), внедрить fail-fast.
2. **P0:** Убрать runtime DDL из приложения, перейти на миграции.
3. **P1:** Ужесточить CORS и токен-стратегию.
4. **P1:** Добавить login rate limiting.
5. **P2:** Стабилизировать тестовый контур и CI.
