# План от текущего этапа до релиза (Release Execution Plan)

Дата: 2026-03-03  
Горизонт: 3–4 недели до production release + 2 недели post-release stabilization.

---

## 1) Точка старта (где мы сейчас)

### 1.1 Готово и уже работает
- Основной контур продукта собран: заказы, оплата, роли, генерация документов, склад/касса номеров, админка, управление аккаунтами.
- UI-часть хедера и меню приведена к более консистентному виду.

### 1.2 Ограничения текущего этапа
- Раздел аналитики выключен/в разработке и в UI отображается как заглушка (в нескольких местах).
- Тестовый прогон в текущем окружении блокируется отсутствием `httpx` (технический долг по окружению/CI, а не по бизнес-логике).

### 1.3 Вывод по стадии
Проект находится на этапе **release candidate hardening**: ядро работает, нужно закрыть операционные, безопасностные и QA-риски, затем выполнить контролируемый релиз.

---

## 2) Цель релиза

Выпустить стабильную версию для ежедневной работы павильонов с предсказуемой эксплуатацией:
- без критичных уязвимостей в конфигурации,
- с воспроизводимым деплоем,
- с понятными регламентами поддержки,
- с базовым набором метрик/логов для диагностики.

---

## 3) Scope релиза (что входит / не входит)

## Входит (MUST)
1. Стабилизация backend-конфига и auth-потока (уже начата, довести до эксплуатационного стандарта).
2. Регламентированный деплой (миграции, env, backup, smoke-check).
3. Тестовый контур (минимум: smoke + auth + orders + payments).
4. Фронтенд-консистентность ключевых экранов (header/menu, таблицы, формы, ошибки).
5. Документация для владельца/оператора + runbook для инцидентов.

## Не входит (defer)
1. Полноценная продуктовая аналитика (если бизнес не требует в этом релизе).
2. Глубокий UX-редизайн всей системы.
3. Крупные архитектурные миграции, не влияющие на текущий SLA.

---

## 4) План работ по этапам

## Этап 0 — Scope freeze и правила релиза (0.5 дня)
**Результат:** зафиксирован состав релиза и критерии «готово».

- Заморозить feature-scope (только bugfix/security/reliability).
- Назначить ответственных: backend, frontend, infra, owner.
- Зафиксировать целевые SLO (доступность, время реакции, время восстановления).

**Критерий завершения:** подписанный checklist релиза.

---

## Этап 1 — Backend/безопасность/конфиг (2–3 дня)
**Результат:** production-safe запуск без «опасных по умолчанию» сценариев.

1. Перепроверить все env-переменные и секреты:
   - `JWT_SECRET` обязателен в production,
   - запрет weak/placeholder значений,
   - убрать любые тестовые креды из deploy-инструкций.
2. Зафиксировать policy CORS:
   - whitelist доменов prod,
   - wildcard только в dev.
3. Проверить auth hardening:
   - лимиты логина,
   - журналирование неудачных попыток,
   - ротация паролей админов.
4. Подтвердить корректность bootstrap-потока суперпользователя.

**Критерий завершения:** security checklist пройден, критичных замечаний нет.

---

## Этап 2 — Деплой и эксплуатация (2 дня)
**Результат:** воспроизводимый и откатный релизный процесс.

1. Зафиксировать production playbook:
   - порядок выкладки,
   - миграции,
   - smoke-тест после деплоя,
   - rollback шаги.
2. Добавить обязательные pre-release операции:
   - backup БД,
   - проверка свободного места,
   - проверка доступности шаблонов документов.
3. Включить structured logging и базовые алерты:
   - 5xx spikes,
   - auth failures spike,
   - DB connection errors.

**Критерий завершения:** dry-run деплоя на staging с успешным rollback rehearsal.

---

## Этап 3 — QA и тест-контур (3–4 дня)
**Результат:** формальный pass по регрессии критического пути.

1. Починить окружение тестов (установка всех test deps, включая `httpx`).
2. Ввести минимальный gate в CI:
   - unit/integration smoke,
   - линт/статические проверки,
   - сборка без ошибок.
3. Пройти ручной UAT-сценарий:
   - логин по ролям,
   - создание/оплата/статусы заказа,
   - печать документов,
   - операции павильона 2 (номера, касса, склад),
   - работа меню/навигации и обработка ошибок на UI.

**Критерий завершения:** test report с pass-rate 100% по критическому пути.

---

## Этап 4 — Frontend polish до релиза (1–2 дня)
**Результат:** целостный, аккуратный интерфейс на всех ключевых экранах.

1. Пройти визуальную матрицу экранов (desktop минимум):
   - index, plate-operator, plate-cash, warehouse, cash-shifts, admin, users, account.
2. Унифицировать состояния компонентов:
   - loading/empty/error/success,
   - фокус и клавиатурная навигация,
   - единая типографика и spacing.
3. Закрыть явные UX-шероховатости в dropdown/таблицах/формах.
4. Сделать финальный screenshot-set «до/после» для владельца.

**Критерий завершения:** визуальный sign-off от владельца.

---

## Этап 5 — Release Candidate и go/no-go (0.5–1 день)
**Результат:** решение о выпуске на production.

Go/no-go meeting (коротко, по чеклисту):
- Security — green
- QA/UAT — green
- Deploy dry-run — green
- Docs/runbook — green
- Backups/rollback readiness — green

**Критерий завершения:** формальное решение GO.

---

## Этап 6 — Production release (день релиза)
**Результат:** релиз в production с контролем рисков.

1. Выполнить backup.
2. Провести деплой по playbook.
3. Выполнить post-deploy smoke-check.
4. Включить усиленный мониторинг на первые 24 часа.
5. Сообщить пользователям о завершении релиза.

**Критерий завершения:** критичный путь пользователя работает без деградации.

---

## Этап 7 — Stabilization (первые 14 дней)
**Результат:** стабилизация после релиза и закрытие хвостов.

1. Ежедневный triage инцидентов/ошибок.
2. Быстрые hotfix-релизы при необходимости (по SLA).
3. Сбор обратной связи операторов и владельца.
4. Формирование backlog vNext (включая аналитику как отдельный трек).

**Критерий завершения:** отсутствие P1/P0 инцидентов в течение 7 дней подряд.

---

## 5) Риски и управление рисками

1. **Риск:** нестабильный тест-контур/зависимости.  
   **Митигация:** зафиксировать dev/test зависимости, воспроизводимое окружение, CI gate.

2. **Риск:** регрессии в auth/ролях.  
   **Митигация:** приоритетные интеграционные тесты + ручной UAT по ролям.

3. **Риск:** проблемы после деплоя.  
   **Митигация:** dry-run, backup перед выкладкой, отработанный rollback.

4. **Риск:** визуальная неоднородность UI.  
   **Митигация:** единая дизайн-матрица экранов и финальный visual sign-off.

---

## 6) Артефакты, которые должны быть к релизу

- Release checklist (подписанный).
- Test report (авто + ручной UAT).
- Deploy playbook + rollback инструкция.
- Incident runbook (что делать при 5xx/auth/db проблемах).
- Changelog релиза и короткая инструкция для сотрудников.

---

## 7) Предложенный календарь (пример)

- **Неделя 1:** Этапы 0–2
- **Неделя 2:** Этапы 3–4
- **Неделя 3:** Этапы 5–6
- **Неделя 4:** Этап 7 (stabilization)

Если команда небольшая, реалистично закладывать +20–30% буфера.

---

## 8) Definition of Done для этого релиза

Релиз считается завершённым, когда одновременно выполнено:
1. Критичный пользовательский путь проходит end-to-end для всех ролей.
2. Нет открытых critical/high багов в release scope.
3. Деплой и rollback воспроизводимы по инструкции.
4. Наблюдаемость и алерты покрывают ключевые инциденты.
5. Владелец дал финальный sign-off.
